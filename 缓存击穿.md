# 第二层：Redis（限流计数状态）

这是更值得讨论的击穿场景。看四种算法的 key 过期行为：

## 固定窗口 — `expire(key, period)`，period 通常是 60 秒

```lua
-- 第 46-49 行
current = redis.call('incr', key)
if tonumber(current) == 1 then
    redis.call('expire', key, period)  -- key 过期后计数重置
end
```

key 过期的瞬间，并发请求同时执行 `INCR`，第一个设置 `EXPIRE`。因为 Lua 脚本保证了原子性，所以这里**不会击穿**。

## 令牌桶 / 漏桶 — expire 设的是 3600 秒（1 小时）

```lua
-- 第 117-118 行（令牌桶）
redis.call('expire', tokens_key, 3600)
redis.call('expire', time_key, 3600)
```

一小时的 TTL 意味着：如果某个接口超过 1 小时没有任何请求，Redis key 过期，状态丢失。下次请求到来时：

```lua
-- 第 105-106 行
local last_tokens = tonumber(redis.call('get', tokens_key)) or capacity  -- key 不存在 → 满桶
local last_time = tonumber(redis.call('get', time_key)) or current_time  -- key 不存在 → 当前时间
```

key 过期后自动回到满桶状态。这在设计上是合理的（长时间没请求理应恢复满额度），**不算击穿问题**。

## 真正存在击穿风险的场景

如果未来把 `RateLimitConfigService` 从全量预热模式改为按需加载模式（cache-aside 模式），击穿问题就会出现：

```java
// ❌ 假设改成了这种写法（按需加载模式）
public RateLimitConfig getActiveConfig(String limitKey) {
    RateLimitConfig config = cache.get(limitKey);
    if (config == null) {
        // 缓存没命中 → 查 DB → 回填缓存
        config = configMapper.findByLimitKey(limitKey);  // ← 击穿点！
        if (config != null) {
            cache.put(limitKey, config);
        }
    }
    return config;
}
```

假设某个热点接口（如 `test:sliding-window`）配置在缓存中过期，每秒有 1000 个并发请求：

```
1000 个请求同时到达
       ↓
cache.get("test:sliding-window") → 全部返回 null
       ↓
1000 个线程同时执行 configMapper.findByLimitKey(...)
       ↓
MySQL 瞬间收到 1000 个相同的 SELECT 查询
       ↓
DB 连接池耗尽 / 响应变慢 / 甚至宕机
```
